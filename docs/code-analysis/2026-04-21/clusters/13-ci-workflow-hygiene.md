# Cluster 13 — ci-workflow-hygiene

## TL;DR

- **Goal:** Run E2E in CI, kill the duplicate Homebrew-tap bump race, decide what to do about the committed generated `assets-embedded.ts`, switch `node node_modules/.bin/playwright` to `bunx playwright`, and delete dead Makefile shell expansions.
- **Impact:** The E2E gap and the Homebrew duplication are real CI hygiene issues; the Homebrew race can silently fail a release run. The others are small polish items that reduce friction.

## Findings

- **E2E tests absent from release CI** — No Playwright step in `release.yml`. `make test` and CLAUDE.md's `act` command both run typecheck + unit + e2e, but the release workflow runs unit only. A browser-layer regression passes CI.
  - Location: `.github/workflows/release.yml` (no Playwright step anywhere in the file)
  - Severity: Medium · Confidence: Verified · Effort: Medium · Autonomy: needs-decision
  - Cluster hint: `ci-hygiene`
  - Raised by: Tooling Analyst
  - Notes: Running Playwright in the full matrix (4 legs × multi-platform) is expensive. Reasonable split: add a single `test-e2e` job on `ubuntu-latest` that runs before the build matrix and gates all build legs. macOS Playwright CI is slow and generally not warranted at T2. The needs-decision is whether linux-only E2E coverage is acceptable (recommended: yes; E2E regressions are mostly DOM/semantic, cross-platform behaviour is already exercised by the unit suite).

- **Duplicate Homebrew tap bump: inline job in `release.yml` + standalone workflow both fire on every release** — `release.yml` contains an inline `homebrew:` job (runs after the `release` job within release.yml). `bump-homebrew-tap.yml` triggers on `release: published`. Because `softprops/action-gh-release` publishes a non-draft release, the second workflow always fires. Both attempt to `git push` the same formula update to the same branch with the same commit message. Under normal timing the second run hits `git diff --quiet` → exit 0 silently; under cold-start the two can run simultaneously and one fails with non-fast-forward, marking the release run red.
  - Location: `.github/workflows/release.yml:150-251` · `.github/workflows/bump-homebrew-tap.yml` (whole file)
  - Severity: Medium · Confidence: Verified · Effort: Small · Autonomy: needs-decision
  - Cluster hint: `release-workflow`
  - Raised by: Tooling Analyst
  - Notes: The simple fix is to remove the inline `homebrew:` job from `release.yml`. The standalone `bump-homebrew-tap.yml` (release: published + workflow_dispatch fallback) is the correct home for this logic — the inline job predates the standalone workflow and is now redundant. No behaviour is lost.

- **`src/server/assets-embedded.ts` is a generated file committed to the repo without a `.gitignore` entry** — Auto-generated by `scripts/generate-assets.ts`, carries a `// @ts-nocheck` header, is committed, and is not in `.gitignore`. `make clean` deletes it; `make build` regenerates it. Committing means every asset change (e.g., a theme edit) produces a diff in both the theme file and `assets-embedded.ts`. The file also references source-relative paths that change when assets are added or removed, producing diff noise.
  - Location: `src/server/assets-embedded.ts:1` · `.gitignore` (no matching entry)
  - Severity: Low · Confidence: Verified · Effort: Small · Autonomy: needs-decision
  - Cluster hint: `ci-hygiene`
  - Raised by: Tooling Analyst
  - Notes: The file is excluded from tsc (`tsconfig.json` exclude list) and uses `@ts-nocheck`. The Makefile `typecheck` target lists it as a prerequisite precisely because CI needs it to exist before typechecking. If gitignored, CI must generate it before typechecking — which already happens in the "Build frontend and generate assets" step. The committed version can drift from what the script would produce if the asset list changes in a branch. Decision: gitignore + always regenerate in CI (currently already the case), or keep committed as a reviewable artifact. Either is defensible.

- **`node node_modules/.bin/playwright` invocation instead of Bun-native runner** — Both `Makefile:45,48` and `package.json:12,14` invoke Playwright via `node node_modules/.bin/playwright test`. Playwright's CLI requires Node internally regardless, so this is technically correct. CLAUDE.md explicitly states "Use `bun`. No `pnpm`, `npm`, `tsx`, or `vitest`." Using `bunx playwright` is the Bun-idiomatic equivalent and is consistent with the `bun x tsc` pattern already used for typecheck.
  - Location: `Makefile:45,48` · `package.json:12,14`
  - Severity: Low · Confidence: Verified · Effort: Small · Autonomy: autofix-ready
  - Cluster hint: `bun-build-robustness`
  - Fix: Replace `node node_modules/.bin/playwright test` with `bunx playwright test` in both files (four occurrences).
  - Raised by: Tooling Analyst
  - Notes: `bunx playwright test` still shells out to Node internally — this is a style/consistency finding only, no functional difference.

- **`PLATFORM` and `ARCH` Makefile variables defined but never referenced** — Makefile lines 1-2 set `PLATFORM` and `ARCH` via `$(shell bun -e ...)`. These are expanded on every `make` invocation (adding two Bun subprocess calls to parse phase) but no recipe or prerequisite references them.
  - Location: `Makefile:1-2`
  - Severity: Low · Confidence: Verified · Effort: Small · Autonomy: autofix-ready
  - Cluster hint: `makefile-simplification`
  - Fix: Delete lines 1-2.
  - Raised by: Tooling Analyst
